# OAuth Login Providers — Design Doc

**Companion to:** [`ADR-003-OAUTH-LOGIN-PROVIDERS.md`](./ADR-003-OAUTH-LOGIN-PROVIDERS.md)
**Status:** Proposed
**Date:** 2026-04-14

This document details the design the ADR locks in. It covers: the provider interface and registry, the PKCE/state store, the website routes, the new compute endpoint, the `account_identities` schema, sequence diagrams for all flows, the cookie model, the sudo gate, and the complete test plan.

## 2. `AuthProvider` interface (final shape)

```ts
// src/lib/auth/providers/types.ts

export type ProviderSlug = "google" | "github";

/** Server-side per-flow context injected by the route handler. */
export interface AuthProviderContext {
  /** Validated post-login destination. Already checked against allow-list. */
  returnTo: string;
  /** PKCE pair generated by the start route. */
  pkce: { verifier: string; challenge: string };
  /** 32-byte url-safe random. */
  state: string;
  /** 32-byte url-safe random. Null for providers that do not support nonce (GitHub). */
  nonce: string | null;
}

/** The identity the adapter returns after a successful callback. */
export interface OAuthIdentity {
  provider: ProviderSlug;
  /** Stable provider user id. Google: id_token.sub. GitHub: user.id as string. */
  providerSub: string;
  /** MUST be verified. Adapter throws `UnverifiedEmailError` if the provider cannot confirm. */
  verifiedEmail: string;
  displayName: string | null;
}

export interface AuthProvider {
  slug: ProviderSlug;

  /** Build the URL to redirect the user-agent to. Never reads state from anywhere else. */
  beginLoginRedirect(ctx: AuthProviderContext): URL;

  /**
   * Validate the callback parameters and exchange them for an identity.
   * MUST throw one of the typed errors below on any failure — never return a partially-populated identity.
   */
  completeLoginFromCallback(
    params: URLSearchParams,
    ctx: AuthProviderContext,
  ): Promise<OAuthIdentity>;
}

// Typed errors — handled uniformly by the route
export class StateMismatchError extends Error {}
export class PkceMismatchError extends Error {}
export class IdTokenInvalidError extends Error { constructor(public claim: string) { super(); } }
export class UnverifiedEmailError extends Error {}
export class ProviderApiError extends Error { constructor(public status: number) { super(); } }
```

### Google adapter sketch (behavioural, not implementation)

- `beginLoginRedirect` → `https://accounts.google.com/o/oauth2/v2/auth` with `response_type=code`, `scope=openid email profile`, `code_challenge_method=S256`, `code_challenge`, `state`, `nonce`, `client_id`, `redirect_uri`, `access_type=online`, `prompt=select_account`.
- `completeLoginFromCallback` → POST `https://oauth2.googleapis.com/token` with `grant_type=authorization_code`, `code`, `code_verifier`, `client_id`, `client_secret`, `redirect_uri`. Parse response. Verify `id_token` via `jose.jwtVerify` against the remote JWKS `https://www.googleapis.com/oauth2/v3/certs`, asserting `iss ∈ { https://accounts.google.com, accounts.google.com }`, `aud === client_id`, `exp` future, `nonce === ctx.nonce`, `email_verified === true`. Return `{ provider: "google", providerSub: claims.sub, verifiedEmail: claims.email, displayName: claims.name ?? null }`.

### GitHub adapter sketch

- `beginLoginRedirect` → `https://github.com/login/oauth/authorize` with `client_id`, `redirect_uri`, `scope=read:user user:email`, `state`, `code_challenge`, `code_challenge_method=S256`, `allow_signup=true`. No `nonce` (not OIDC).
- `completeLoginFromCallback` → POST `https://github.com/login/oauth/access_token` with `client_id`, `client_secret`, `code`, `redirect_uri`, `code_verifier`, `Accept: application/json`. Then GET `https://api.github.com/user` with `Authorization: Bearer <access_token>`, `User-Agent: mmpm-website`. Then GET `https://api.github.com/user/emails` and find the first `{ verified: true, primary: true }` entry. If none, throw `UnverifiedEmailError`. Return `{ provider: "github", providerSub: String(user.id), verifiedEmail: primary.email, displayName: user.name ?? user.login }`. Discard the access token.

## 3. PKCE / state store

An in-process `Map<state, Entry>` with TTL for single-replica dev, and a Redis-backed equivalent in prod if we scale horizontally. Entry:

```ts
interface PkceEntry {
  provider: ProviderSlug;
  verifier: string;
  nonce: string | null;
  returnTo: string;
  createdAt: number;         // ms
  used: boolean;             // single-use guard
}
```

Rules:

- `put(state, entry)` → inserts with `createdAt = Date.now()`.
- `consume(state)` → atomic read+mark-used+return. If `used || (now - createdAt) > 5 * 60_000`, returns `null`. Caller must treat `null` as `StateMismatchError`.
- Background sweep every 60 s removes entries older than 10 minutes (belt-and-braces). Note: single TTL of 5 min covers both the `state` binding and the PKCE verifier — they share this store entry, so the stricter recommendation (state ≤ 5 min) wins.
- The `state` value itself is 32 bytes of `crypto.getRandomValues` encoded as base64url. Collision-free and unguessable.

## 4. Cookies

| Cookie              | Purpose                                    | Flags                                         | Lifetime |
|---------------------|--------------------------------------------|-----------------------------------------------|----------|
| `mmpm_session`      | Bearer into compute                        | `HttpOnly; Secure; SameSite=Lax; Path=/`      | 30 d     |
| `mmpm_oauth_state`  | Pre-flight binding for OAuth state         | `HttpOnly; Secure; SameSite=Lax; Path=/api/auth/oauth` | 5 min |
| `mmpm_sudo`         | Elevated session for linking/unlinking     | `HttpOnly; Secure; SameSite=Lax; Path=/api/auth/oauth` | 5 min  |

Why `SameSite=Lax` for `mmpm_oauth_state`: the OAuth callback is a top-level navigation from `accounts.google.com` / `github.com` back to us. Lax allows top-level navigations; Strict would drop the cookie on the callback and break the flow.

Why `Path=/api/auth/oauth` on `mmpm_oauth_state` and `mmpm_sudo`: minimises exposure. Regular page loads and compute proxy calls never see these cookies.

## 5. Routes — website

### `POST /api/auth/oauth/:provider/start`

Input (form body or JSON):
```json
{ "returnTo": "/dashboard" }
```

Logic:
1. Validate `:provider` against the registry. 404 on unknown.
2. Validate `returnTo` against the allow-list. `returnTo` must begin with `/` (single leading slash, no `//`), be an ASCII path, and match `^/(dashboard|admin|account|auth-methods|$)` (adjust as needed). Reject otherwise.
3. Generate `state`, `nonce` (if provider is OIDC), `pkce.{verifier, challenge}`.
4. Store a `PkceEntry` in the store keyed by `state`.
5. Set `mmpm_oauth_state` cookie to the same `state`.
6. Call `provider.beginLoginRedirect(ctx)` and `302` to the returned URL.

Returns `302` → provider.

### `GET /api/auth/oauth/:provider/callback`

Inputs: query string with `code`, `state`, optionally `error`, `error_description`.

Logic:
1. If `error` present → log `{ provider, outcome: "provider_error", code: error }`, redirect to `/login?error=provider_denied`.
2. Read cookie `mmpm_oauth_state`. If missing or does not match query `state` → `400 invalid_state`.
3. `consume(state)` from the store. If `null` → `400 invalid_state`. If present, copy `verifier`, `nonce`, `returnTo` from the entry, then the entry is gone.
4. Call `provider.completeLoginFromCallback(params, ctx)`. Any thrown error → log with opaque reason, `302 /login?error=<opaque>`.
5. Call compute bridge: `POST {COMPUTE_URL}/api/auth/oauth/complete` with `X-Compute-Bridge: <secret>` and the `OAuthIdentity`. Expect `{ sessionToken, accountId }`.
6. **Rotate**: clear `mmpm_oauth_state`, set `mmpm_session` to the new `sessionToken`, `302` to `returnTo`.
7. Emit log entry: `{ event: "oauth_signin", provider, outcome: "success", user_id: accountId, ip, ua, ts }`.

### `POST /api/auth/oauth/link`

Adds a provider to the currently logged-in user. Requires a valid sudo session.

Input: `{ "provider": "google" | "github" }` — triggers a normal OAuth start flow but marks the `PkceEntry` with `intent: "link"`. The callback then POSTs to `POST {COMPUTE_URL}/api/auth/oauth/link` (new compute route) with the `X-Compute-Bridge` token instead of calling `/oauth/complete`. The link route refuses if the target provider is already linked to this account, or if the provider sub is already linked to a different account.

### `POST /api/auth/oauth/unlink`

Input: `{ "provider": "google" | "github" }`. Sudo-gated. Compute refuses to delete the last remaining identity of an account (to prevent the user from locking themselves out).

## 6. Compute — new endpoint

### `POST /api/auth/oauth/complete`

**Authentication:** server-to-server only. Requires `X-Compute-Bridge: <COMPUTE_OAUTH_BRIDGE_TOKEN>`. No bearer session accepted. Missing/wrong header → `401`, no body. Never logs the token.

**Request:**
```json
{
  "provider": "google" | "github",
  "provider_sub": "string",
  "verified_email": "string",
  "display_name": "string | null"
}
```

**Response (200):**
```json
{
  "ok": true,
  "sessionToken": "<raw 256-bit hex>",
  "accountId": "<uuid>"
}
```

**Error responses:** `400` on schema failure; `401` on bad bridge token; `409` if `(provider, provider_sub)` is already linked to a different account (consistency guard — should be impossible unless the website is buggy).

**Transactional logic (single Postgres transaction):**

```
BEGIN;
  SELECT account_id FROM account_identities
   WHERE provider = $1 AND provider_sub = $2;
  -- case A: identity found → account_id = found; UPDATE last_used_at.
  -- case B: identity not found →
  --   SELECT id FROM accounts WHERE email = $3;
  --   case B1: existing account → INSERT INTO account_identities (...).
  --   case B2: no account → INSERT INTO accounts (...) RETURNING id;
  --                         (same Stripe customer provisioning as verifyMagicLink)
  --                         INSERT INTO account_identities (...).
  -- In all cases: INSERT INTO auth_sessions (token_hash, account_id, ...);
COMMIT;
-- Return the raw token (NOT the hash) to the website.
```

The whole operation is a single DB transaction so "find or create account + identity + session" is atomic. The Stripe customer creation is the only side effect that is not rolled back by a DB rollback — that is pre-existing behaviour from `verifyMagicLink` and is not made worse by this change. (A follow-up can introduce a `signup_events` outbox table to make Stripe provisioning idempotent, but that is ADR-0xx, not this ADR.)

### `POST /api/auth/oauth/link`

Same authentication as `/oauth/complete`, but receives an extra `account_id` from the website (the currently-logged-in user) and refuses if:

- `(provider, provider_sub)` is already linked to any account;
- `(account_id, provider)` is already linked.

Both checks are enforced by the `UNIQUE` constraints — the handler just translates the `23505` SQLSTATE into a semantic `409 conflict`.

### `POST /api/auth/oauth/unlink`

Receives `{ account_id, provider }`. Refuses if the target account would be left with zero `account_identities` rows AND zero magic-link-only access (i.e., no `accounts.email` match path remaining). For the initial pass — while magic link is not yet represented in `account_identities` — compute treats "has a verifiable email" as an implicit always-available fallback and allows unlinking any OAuth provider. Document this clearly and revisit when magic link is migrated into the identities table.

## 7. Sequence diagrams

### A. New user signs in with Google (happy path)

```
Browser           Website (/api/auth/oauth/google/...)     Google          Compute
  |                           |                              |               |
  |--POST /start{returnTo}--->|                              |               |
  |                           | gen state+nonce+PKCE         |               |
  |                           | store PkceEntry              |               |
  |                           | Set-Cookie: mmpm_oauth_state |               |
  |<---302 Location: accounts.google.com/o/oauth2/v2/auth?... |               |
  |                                                          |               |
  |--GET /o/oauth2/v2/auth------------------------------------>               |
  |                                                          |               |
  |<--302 Location: /api/auth/oauth/google/callback?code=...&state=...--------|
  |                                                          |               |
  |--GET /callback?code=&state=--->|                         |               |
  |                           | verify cookie state == query |               |
  |                           | consume(state) → entry       |               |
  |                           |---POST /token (code+verifier)->|              |
  |                           |<--access_token+id_token-------|               |
  |                           | jose.jwtVerify(id_token, JWKS)|               |
  |                           | assert iss/aud/exp/nonce/email_verified     |
  |                           |---POST /api/auth/oauth/complete -------------->|
  |                           |   X-Compute-Bridge: ***                      |
  |                           |   {provider,sub,email,name}                  |
  |                           |                                              | BEGIN
  |                           |                                              | SELECT identity → none
  |                           |                                              | SELECT accounts.email → none
  |                           |                                              | INSERT account
  |                           |                                              | provision Stripe
  |                           |                                              | INSERT identity
  |                           |                                              | INSERT auth_sessions
  |                           |                                              | COMMIT
  |                           |<--{ok,sessionToken,accountId}----------------|
  |                           | clear mmpm_oauth_state cookie                 |
  |                           | Set-Cookie: mmpm_session=<sessionToken>      |
  |<--302 Location: /dashboard|                                              |
```

### B. Existing magic-link user signs in with Google (auto-link)

Same as A except in the compute transaction:

```
BEGIN
SELECT identity → none
SELECT accounts.email → exists  (existing magic-link user)
INSERT account_identities (account_id, 'google', sub, email)
INSERT auth_sessions
COMMIT
```

The user lands on `/dashboard` signed in as their existing account, with a Google identity now attached. Next time they click "Sign in with Google" → case where identity is found.

```
Browser       Website                                     Compute
  |              |                                           |
  |--POST /api/auth/oauth/link{provider:github}-->|          |
  |              | session cookie valid? yes                 |
  |              | sudo cookie valid? NO                     |
  |<-- 403 {error:"sudo_required"} ----|
  |                                                          |
  |  -- user re-authenticates via existing magic link --     |
  |  (reuses existing /request-link + /verify + sudo mint)   |
  |                                                          |
  |--POST /api/auth/oauth/link{provider:github}-->|          |
  |              | sudo cookie valid? yes                    |
  |              | generate state+PKCE (intent=link)         |
  |              | store PkceEntry { intent: link,           |
  |              |                   accountId: currentUser }|
  |              | Set-Cookie: mmpm_oauth_state              |
  |<--302 github.com/login/oauth/authorize?...      |        |
  |  -- user completes GitHub auth --                        |
  |--GET /callback?code=&state=--->|                         |
  |              | validate state, PKCE, fetch /user+/user/emails
  |              |---POST /api/auth/oauth/link ----------------->|
  |              |  X-Compute-Bridge, account_id, identity       |
  |              |                                              | INSERT identity
  |              |                                              | (UNIQUE constraint enforces no duplicate)
  |              |<--200 {ok}-----------------------------------|
```

### D. Attack path: unverified email — must fail

```
Attacker  Website                                          Google (mock)
  |         |                                                 |
  |  ... start flow normally ...                              |
  |         |<---code (attacker's code)-----------------------|
  |         | POST /token                                     |
  |         |<---id_token { email_verified: false }-----------|
  |         | jose.jwtVerify passes (signature valid)         |
  |         | assertion: email_verified === true → FAIL       |
  |         | throw UnverifiedEmailError                      |
  |<-- 302 /login?error=unverified_email                      |
```

No call to compute is ever made. `account_identities` unchanged. Log entry: `{ event: "oauth_signin", outcome: "fail", reason: "unverified_email" }`.

## 9. Operational concerns

- **Env vars (new):** `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET`, `GOOGLE_OAUTH_REDIRECT_URI`, `GITHUB_OAUTH_CLIENT_ID`, `GITHUB_OAUTH_CLIENT_SECRET`, `GITHUB_OAUTH_REDIRECT_URI`, `COMPUTE_OAUTH_BRIDGE_TOKEN`, `NEXT_PUBLIC_OAUTH_ENABLED`. Separate values for dev and prod — see `.env.example` update.
- **OAuth app registration:** create one Google OAuth 2.0 Client ID and one GitHub OAuth App per environment (dev, staging, prod). Register exact redirect URIs. Do NOT use wildcards. Do NOT share client IDs across environments.
- **Rate limiting:** the existing per-IP/per-email limiter on `/api/auth/*` is extended to cover `/api/auth/oauth/*`. Reuse the existing middleware — don't introduce a second limiter.
- **Logging schema:** add to structured logger a new event family `oauth_*`. Fields: `event`, `provider`, `outcome`, `reason`, `account_id` (post-link only), `ip`, `ua`, `ts`. Assert in CI that no OAuth token, PKCE verifier, ID token, or bridge secret appears anywhere in stdout/stderr.
- **Feature flag:** `NEXT_PUBLIC_OAUTH_ENABLED` gates the sign-in buttons. Server-side routes check a server-side counterpart `OAUTH_ENABLED` (distinct so the client flag can be enabled after the server is ready). Both must be true to advertise and to accept.
- **Observability dashboards:** add panels for `oauth_signin{provider=google, outcome=success}`, `{outcome=fail}`, `oauth_link` events, per-provider 4xx rates on `/callback`. Alerts on: >5% callback error rate sustained over 10 min, or any `privileged: true` auto-link (see N6).

---

## 10. Migration order (condensed)

1. Compute: add migration `NNN_account_identities.sql`; add `POST /api/auth/oauth/complete`, `link`, `unlink` handlers; add `COMPUTE_OAUTH_BRIDGE_TOKEN` env. Deploy. Unit+integration tests on compute only. **No user-visible change.**
2. Website: add `AuthProvider` interface + adapters + PKCE store + new routes behind `NEXT_PUBLIC_OAUTH_ENABLED=false`. Deploy to staging. Integration + E2E tests.
3. Security review: run the full negative suite, verify the go/no-go checklist.
4. Flip `NEXT_PUBLIC_OAUTH_ENABLED=true` in prod. Watch logs for 72h.
5. (Later) Refactor magic link into the `AuthProvider` interface, backfill `account_identities` with `provider='magic_link'` rows, retire `magic_link_tokens` table 30 days after.

## 11. Open questions for the team

1. Is the `COMPUTE_OAUTH_BRIDGE_TOKEN` approach acceptable as a first pass, or do we want mTLS between website and compute from day one? (mTLS is cleaner, costs a one-time cert setup.)
2. For admin-account auto-link (test N6), do we want to **block** auto-link when the target account has elevated privileges, or just **alert**? Blocking is safer; alerting is simpler.
3. When a logged-in user unlinks their only remaining OAuth provider and the account has no magic-link history (N/A today, relevant once magic link is migrated into identities), what should compute return? (Suggestion: 409 with `error: "last_auth_method"`.)
4. Do we want to store the access token for GitHub for any future integration use-case? (Default: no. Smaller attack surface.)
